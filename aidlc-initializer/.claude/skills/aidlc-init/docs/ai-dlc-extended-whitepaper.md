# AI-DLC 확장 백서: 프로젝트 분석 기반 적용

**원본 저자:** Raja SP, Amazon Web Services
**확장:** 이 문서는 원본 AI-DLC 백서(`ai-dlc-whitepaper-ko.md`)를 기반으로 합니다.

---

## I. 서문

본 문서는 Raja SP(AWS)가 정의한 AI 주도 개발 라이프사이클(AI-DLC) 방법론의 확장 해석입니다. 원본 백서는 AI-DLC의 핵심 철학과 원칙을 명확히 정립했으며, 그 가치는 변함이 없습니다. 다만 원본이 도메인 주도 설계(DDD) 변형과 AWS 인프라를 중심으로 설명되어 있어, 다양한 프로젝트 유형을 다루는 실무자에게는 적용 방식에 대한 추가적인 고민이 필요했습니다.

이 확장 백서의 목적은 새로운 방법론을 정립하는 것이 아닙니다. AI-DLC의 핵심 철학—대화 방향의 역전, 볼트 사이클, 계획-승인-실행 패턴—을 온전히 유지하면서, 프로젝트의 특성을 먼저 분석하고 그에 적합한 접근방식을 선택하는 과정을 설명합니다.

원본 백서를 먼저 읽으시기를 권합니다. 이 문서는 원본의 보완재이지 대체재가 아닙니다.

---

## II. 원본 백서에 대한 고찰

### 핵심 철학의 유효성

원본 AI-DLC 백서가 제시하는 핵심 원칙들은 AI와 함께 복잡한 문제를 해결하는 모든 프로젝트에 적용 가능합니다.

**대화 방향의 역전**은 인간이 AI에게 작업을 지시하는 전통적 패러다임에서 벗어나, AI가 계획을 제안하고 인간이 검증하는 방식으로의 전환을 의미합니다. 이는 어떤 유형의 프로젝트에서도 동일하게 적용됩니다.

**볼트 사이클**은 시간 또는 일 단위의 빠른 반복을 통해 지속적인 검증과 피드백을 가능하게 합니다. 주 단위 스프린트에 비해 훨씬 짧은 이 사이클은 AI의 빠른 생성 능력과 인간의 판단력을 효과적으로 결합합니다.

**계획-승인-실행 패턴**은 AI가 먼저 계획을 수립하고, 인간이 이를 검토하여 승인한 후에야 실행이 이루어지는 구조입니다. 이 패턴은 AI의 자율성과 인간의 통제권 사이에서 적절한 균형점을 찾습니다.

이러한 철학들은 특정 기술이나 방법론과 무관하게 AI와 인간이 협업하는 방식 자체를 정의합니다. 현존하는 모든 검증된 접근방식을 사용할 수 있습니다. AI-DLC가 정의하는 것은 AI와 인간 사이의 커뮤니케이션, 결정, 작업 프로세스에 대한 철학입니다.

### 특정 접근방식에 대한 고려

원본 백서는 DDD(도메인 주도 설계) 변형을 중심으로 설명하며, AWS 인프라 컴포넌트를 예시로 사용합니다. 이는 복잡한 엔터프라이즈 시스템을 구축할 때 매우 유효한 접근방식입니다. 그러나 모든 프로젝트가 이러한 수준의 복잡도를 필요로 하지는 않습니다.

예를 들어, 새로운 언어나 DSL을 설계할 때는 문법 정의가 핵심이 되고, 자동화 파이프라인을 구축할 때는 단계와 흐름의 설계가 중심이 됩니다. 분석 보고서나 문서를 체계적으로 생성하는 작업, 기존 시스템을 전면 개편하는 작업, 콘텐츠 생성 시스템을 구축하는 작업 등 다양한 유형의 프로젝트가 있습니다. 이러한 프로젝트들에서 DDD의 모든 개념이 필요하지 않을 수 있습니다. 프로젝트의 성격에 따라 적합한 접근방식은 다양합니다.

중요한 것은 AI-DLC의 철학이지, 특정 설계 방법론이나 기술이 아닙니다.

## III. 프로젝트 분석 기반 적용

### 핵심 원칙

AI-DLC를 적용할 때 가장 먼저 해야 할 일은 프로젝트를 분석하는 것입니다. 프로젝트의 특성을 이해한 후에야 어떤 접근방식이 적합한지 판단할 수 있습니다.

이 과정에서 AI가 프로젝트 분석을 주도하고, 인간이 그 분석 결과를 검증합니다. 이것이 바로 대화 방향의 역전입니다. AI가 "이 프로젝트는 이러이러한 특성을 가지고 있으므로, 이러한 접근방식을 권장합니다"라고 제안하면, 인간은 이를 검토하고 승인하거나 수정 피드백을 제공합니다.

### 분석 과정

프로젝트 분석은 다음과 같은 요소들을 고려합니다.

**프로젝트 상태**를 파악합니다. 빈 폴더에서 시작하는 신규 프로젝트인지, 이미 존재하는 것을 개선하는 것인지, 기존 시스템에 새로운 것을 추가하는 것인지에 따라 접근 방식이 달라집니다.

**프로젝트 성격**을 파악합니다. 무엇을 만들고자 하는지, 어떤 문제를 해결하고자 하는지, 최종 산출물은 무엇인지 이해합니다.

**사용할 도구와 환경**을 확인합니다. 이미 선택된 것이 있다면 그것을 기반으로 진행합니다. 아직 결정되지 않았다면, 프로젝트 요구사항에 맞게 함께 선정합니다.

**복잡도**를 평가합니다. 문제의 복잡도, 관련된 이해관계자, 제약사항, 의존성 등을 고려합니다.

이러한 분석을 통해 AI는 프로젝트에 적합한 접근방식을 권장합니다. 프로젝트의 성격에 따라 적합한 방법은 다양합니다.

### 접근방식의 선택

AI-DLC는 특정 방법론을 강제하지 않습니다. 원본 백서의 원칙 3 "설계 기술을 핵심으로 통합"은 체계적 접근의 중요성을 강조할 뿐, 특정 방법론만을 유일한 선택지로 제시하지 않습니다.

중요한 것은 선택한 접근방식을 AI-DLC의 워크플로 안에서 일관되게 적용하는 것입니다. AI가 해당 접근방식에 따라 계획을 수립하고, 인간이 이를 검증하며, 승인 후 단계적으로 실행합니다.

## IV. 프롬프트 생성

### 프로젝트에 맞는 프롬프트

원본 백서의 부록 A는 AI-DLC 실습을 위한 프롬프트 예시를 제공합니다. 이 프롬프트들은 훌륭한 출발점이지만, 모든 프로젝트에 그대로 적용하기보다는 프로젝트 분석 결과에 맞게 조정되어야 합니다.

프로젝트의 성격에 따라 프롬프트에서 다루는 용어와 개념이 달라집니다. 언어를 설계하는 프로젝트라면 문법과 파서를 다루는 프롬프트가 필요하고, 자동화 파이프라인을 구축하는 프로젝트라면 단계와 흐름을 다루는 프롬프트가 필요합니다. 문서를 생성하는 프로젝트라면 구조와 품질 기준을 다루는 프롬프트가 필요합니다.

### 프롬프트 구조

AI-DLC 프롬프트는 공통된 구조를 따릅니다.

**역할 정의**로 시작합니다. AI에게 수행할 역할을 부여합니다.

**계획 수립 지시**가 따릅니다. 작업을 시작하기 전에 계획을 세우고, 그 계획을 파일에 기록하도록 합니다. 이는 AI-DLC의 핵심인 계획-승인-실행 패턴을 구현합니다.

**승인 요청**을 명시합니다. "계획을 생성하면 제 검토와 승인을 요청하세요. 제 승인 후에만 작업을 진행할 수 있습니다."

**작업 정의**로 마무리합니다. 구체적으로 수행해야 할 작업을 설명합니다.

어떤 단계든 명확히 해야 할 부분이 있다면 질문을 추가하고 응답을 요청합니다. 이는 AI와 인간 사이의 원활한 대화를 위한 규약입니다. 질문과 응답의 형식은 사용하는 환경에 따라 두 가지 패턴을 사용할 수 있습니다.

**태그 패턴**은 범용적으로 사용할 수 있는 방식입니다. [Question] 태그와 함께 질문을 작성하고, 응답을 채울 수 있도록 빈 [Answer] 태그를 생성합니다. 이 패턴은 어떤 AI 도구에서도 복사하여 사용할 수 있습니다.

**Claude Code 패턴**은 Claude Code CLI 환경에서 사용하는 방식입니다. AskUserQuestion 툴을 사용하여 질문하고, 사용자로부터 직접 응답을 받습니다. 이 패턴은 더 나은 상호작용 경험을 제공하지만, Claude Code 환경에서만 사용할 수 있습니다.

## V. AI-DLC 단계의 적용

### 구상 단계 (Inception)

구상 단계의 목적은 의도를 명확히 하고 이를 실행 가능한 작업 단위로 분해하는 것입니다. 이 목적은 프로젝트 유형과 무관하게 동일합니다.

프로젝트의 성격에 따라 분해 방식은 달라집니다. 무엇을 만들고자 하는지, 어떤 문제를 해결하고자 하는지에 따라 적절한 단위로 나눕니다.

원본 백서의 "모브 엘라보레이션" 의식은 여기서도 유효합니다. AI가 분해를 제안하고, 인간이 검증하며, 함께 세련화합니다.

### 구축 단계 (Construction)

구축 단계는 정의된 작업을 실제로 수행합니다. 코드를 작성하는 것일 수도 있고, 문서를 생성하는 것일 수도 있으며, 시스템을 구성하는 것일 수도 있습니다. 수행 방식은 프로젝트에 선택된 접근방식을 따릅니다.

중요한 것은 각 단계에서 AI가 먼저 제안하고, 인간이 검증한 후 다음 단계로 진행한다는 것입니다. 이것이 볼트 사이클입니다.

### 운영 단계 (Operations)

운영 단계의 원칙도 동일하게 적용됩니다. AI가 검증, 적용, 유지보수를 위한 권장사항을 제시하고, 인간이 이를 검증하고 승인합니다.

산출물이 코드든, 문서든, 시스템 구성이든, AI-DLC의 워크플로는 변하지 않습니다.

## VI. 실천 가이드

### 시작하기

AI-DLC를 프로젝트에 적용하려면 다음 단계를 따릅니다.

먼저 프로젝트 분석을 수행합니다. 프로젝트의 상태, 성격, 도구, 복잡도를 파악하고, 적합한 접근방식을 선정합니다. 이 과정에서 AI가 분석을 주도하고 인간이 검증합니다.

다음으로 프롬프트 세트를 생성합니다. 분석 결과를 바탕으로 프로젝트에 맞는 AI-DLC 프롬프트를 작성합니다. 원본 백서의 예시를 참고하되, 프로젝트 특성에 맞게 조정합니다.

그리고 AI-DLC 워크플로를 따라 진행합니다. 구상, 구축, 운영 단계를 거치며, 각 단계에서 계획-승인-실행 패턴을 유지합니다.

### 유의사항

AI-DLC는 도구가 아니라 철학입니다. 특정 환경이나 AI 서비스에 종속되지 않습니다. 중요한 것은 AI와 인간이 협업하는 방식입니다.

프로젝트마다 적합한 접근방식은 다릅니다. 하나의 접근방식을 모든 프로젝트에 강제하지 마십시오. 프로젝트를 먼저 분석하고, 그에 맞는 접근방식을 선택하십시오.

원본 백서의 핵심 원칙들을 항상 염두에 두십시오. 대화 방향의 역전, 볼트 사이클, 계획-승인-실행 패턴. 이것들이 AI-DLC의 본질입니다.

---

## VII. 결론

AI-DLC의 가치는 특정 기술이나 방법론에 있지 않습니다. AI와 인간이 협업하여 복잡한 문제를 해결하는 방식을 정의한 데 있습니다.

이 확장 백서는 그 핵심 가치를 유지하면서, 다양한 프로젝트에 AI-DLC를 적용할 수 있도록 프로젝트 분석 기반 접근방식을 제안했습니다. 프로젝트를 먼저 이해하고, 그에 맞는 접근방식을 선택하며, AI-DLC의 워크플로를 따라 진행하는 것입니다.

원본 백서와 함께 이 문서가 여러분의 AI 주도 작업에 도움이 되기를 바랍니다.

## 부록: 프로젝트 분석 프롬프트

다음은 프로젝트 분석을 시작하기 위한 기본 프롬프트입니다. 사용하는 환경에 따라 태그 패턴 또는 Claude Code 패턴을 선택합니다.

### 태그 패턴 (범용)

```
당신의 과제: AI-DLC 방법론을 적용하기 위해 프로젝트를 분석하고 적합한 접근방식을 선정합니다.

먼저 프로젝트에 대해 알려주세요.

[Question]
프로젝트의 현재 상태는 무엇인가요?
- 빈 폴더에서 시작하는 신규 프로젝트
- 이미 존재하는 것을 개선하는 작업
- 기존 시스템에 새로운 것을 추가하는 작업
[/Question]

[Answer]
{응답 대기}
[/Answer]

[Question]
프로젝트에 대해 설명해주세요.
- 무엇을 만들거나 해결하고자 하나요?
- 최종 산출물은 무엇인가요?
- 어떤 도구나 환경을 사용하나요?
- 특별한 제약사항이나 고려사항이 있나요?
[/Question]

[Answer]
{응답 대기}
[/Answer]
```

### Claude Code 패턴 (Claude Code CLI 전용)

```
당신의 과제: AI-DLC 방법론을 적용하기 위해 프로젝트를 분석하고 적합한 접근방식을 선정합니다.

먼저 AskUserQuestion 툴을 사용하여 프로젝트에 대한 정보를 수집합니다.

첫 번째 질문으로 프로젝트의 현재 상태를 선택받습니다. 신규 프로젝트, 개선 작업, 기능 추가 중 하나를 선택하도록 합니다.

두 번째 질문으로 프로젝트 설명을 입력받습니다. 무엇을 만들거나 해결하고자 하는지, 최종 산출물은 무엇인지, 어떤 도구나 환경을 사용하는지, 특별한 제약사항이 있는지 등을 자유롭게 입력받습니다.
```

응답을 바탕으로 AI는 프로젝트 특성을 정리하고, 적합한 접근방식을 권장하며, 이를 보고서로 작성합니다. 인간이 보고서를 검토하고 승인하면, AI-DLC 프롬프트 세트 생성으로 진행합니다.
